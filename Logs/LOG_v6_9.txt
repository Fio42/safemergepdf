PATRÓN PYODIDE INIT CORRECTO — OBLIGATORIO EN TODA TOOL (v6.9)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
PROBLEMA QUE CAUSABA "Loading engine" infinito:
  - pyodide se asignaba ANTES de que pyReady=true
  - Si fallaba después del await loadPyodide() pero antes de pyReady=true,
    la variable pyodide quedaba asignada y la guard if(pyodide||pyReady)
    impedía reintentar → loop infinito de carga

SOLUCIÓN: usar tres flags separados + never check pyodide variable as guard:

  let pyodide = null, pyReady = false, pyLoading = false;

  async function initPyodide() {
    if (pyReady) { /* habilitar botón */ return; }
    if (pyLoading) return;   // ya está cargando, esperar
    pyLoading = true;
    /* mostrar spinner */
    try {
      pyodide = await loadPyodide({...});
      await pyodide.loadPackage('micropip');
      await pyodide.runPythonAsync(`import micropip; await micropip.install([...])`);
      pyReady = true;
      pyLoading = false;
      /* ocultar spinner, habilitar botón */
    } catch(e) {
      pyLoading = false;
      pyodide = null;   // reset para permitir reintento
      /* mostrar error con instrucción de refresh */
    }
  }

  // En handleFile(): si pyReady → habilitar botón directamente (sin llamar initPyodide)
  // si !pyReady → llamar initPyodide()

REGLA: NUNCA usar `if (pyodide || pyReady)` como guard. Solo `if (pyReady)` + `if (pyLoading)`.

1. COMPRESS-PDF — BUG: Al comprimir un archivo y luego cargar otro,
   el botón Compress quedaba disabled para siempre.
   CAUSA: resetFull() ponía disabled=true, handleFiles() llamaba
   initPyodide() que hacía early return (engine ya cargado) sin
   re-habilitar el botón.
   FIX: En handleFiles(), si pyodideReady → habilitar botón inmediatamente.
   BONUS: setQuality() ahora re-ejecuta compresión si ya hay resultado
   mostrado → cambiar calidad actualiza el resultado automáticamente.
   BONUS: botón "Compress another file" en result area.
   BONUS: guard corregido a pyLoading para evitar race conditions.

2. PROTECT-PDF — BUG: "Loading engine" se mostraba y nunca desaparecía.
   CAUSA: race condition — pyodide se asignaba antes de pyReady=true,
   si el proceso fallaba pyodide != null pero pyReady = false → loop.
   FIX: patrón pyLoading, reset pyodide=null en catch, mejor mensaje de error.
   FIX EXTRA: si engine ya está listo al cargar nuevo archivo, no
   muestra spinner ni llama initPyodide de nuevo.

3. UNLOCK-PDF — mismo bug y fix que protect-pdf.

4. PDF-TO-JPG (modo JPG→PDF) — BUG: "Loading engine" infinito.
   CAUSA: mismo race condition en initPyodide.
   FIX: patrón pyLoading, re-throw de error para que j2pConvert lo maneje,
   mensaje de error claro al usuario si falla.
